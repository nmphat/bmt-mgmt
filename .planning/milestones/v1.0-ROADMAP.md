**Status:** SHIPPED 2026-06-02

**Total Plans:** 15  
**Total Tasks:** 37  
**Audit:** Passed, 29/29 requirements, 11/11 integration flows

## Overview

v1.0 refactored the Badminton Session Manager into a mobile-first Vue/Supabase SPA while preserving the live backend contract. The milestone focused on debt-first guest payments, role-aware mobile navigation, session-detail task sections, admin/supporting screens, payment dialog polish, and no-regression evidence across routes, Supabase contracts, i18n, build, and browser smoke.

The work used Open Design as directional UI input only. The live app remained the source of truth for routes, permissions, Supabase RPCs/views/tables, payment semantics, attendance behavior, realtime/polling, and localization.

## Milestone Summary

**Key accomplishments:**

1. Delivered a preservation-first mobile shell, debt-first home/member flows, and QR payment UX for guests and authenticated users.
2. Refactored session detail into task sections while preserving attendance, cost summaries, payment snapshots, realtime/polling, admin gates, and public read-only access.
3. Polished sessions, create-session, member management, QR/manual payment surfaces, and table-to-card mobile layouts.
4. Restored DB-backed bank settings and active-bank QR generation through Supabase-backed configuration.
5. Added browser-harness smoke coverage with categorized screenshots and fixed route/navigation/UI regressions found during sweeps.
6. Closed post-merge backend-contract audit gaps for `create_session_with_bookings`, removed `is_permanent`, and current `sessions.update` fields.

**Key decisions:**

- Open Design is a directional reference, not an implementation source of truth.
- Frontend must display Supabase RPC/view results and must not recalculate shared fees.
- Guest debt/payment flows stay public; admin create/edit/manage actions stay protected by route, UI, and handler gates.
- `/sessions` remains public read-only while create/mutation controls remain admin-only.
- Human visual UAT was explicitly skipped/deferred; automated source/build/browser evidence substituted for milestone verification.
- Mark-absent behavior now uses `interval_presence` as source of truth after the removed registration flag was merged out of the schema.

**Issues resolved during milestone:**

- Session-detail admin mutation gates tightened from authenticated-only to admin-only.
- In-flight session refreshes no longer drop realtime/polling/manual/payment-complete refresh events.
- `Tiền sân NaN ₫` display regression fixed and retested.
- Safe-area Tailwind `calc()` CSS warning resolved by moving calculations to CSS helpers.
- Member-detail session history rows/cards now navigate to session detail while QR buttons keep modal behavior.
- Final audit blockers closed: create-session RPC mismatch, member CRUD removed-column mismatch, and session edit non-table field mismatch.

**Issues deferred:**

- Full authenticated admin mutation smoke still requires explicit admin credentials and preferably safe dev/test data.
- Optional low-priority hardening remains for non-color cues in dense financial rows/cards.
- Optional formal Nyquist validation can still be run for Phase 3 if strict Nyquist metadata is required.

**Technical debt noted:**

- Some historical phase artifacts still reference earlier schema contracts; the milestone audit documents current-source closure and supersedes those historical checks.
- Payment polling has two implementation paths (`check_qr_status` composable and direct snapshot polling in `PaymentQRModal.vue`).
- Some profile/bank-config error strings remain hardcoded Vietnamese.
