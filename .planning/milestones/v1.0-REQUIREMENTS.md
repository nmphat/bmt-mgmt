# Requirements Archive: Badminton Session Manager v1.0 UI Refactor

**Status:** Archived as complete  
**Archived:** 2026-06-02  
**Audit:** Passed, 29/29 requirements satisfied  
**Core Value:** Members and guests can understand what they owe and pay it quickly, while admins can manage sessions without redoing spreadsheet work.

## Requirement Outcomes

All v1.0 requirements were implemented and verified. The milestone audit initially found three backend-contract gaps after PR #2 merged; those gaps were closed on `fix/v1-audit-gaps` and the final audit status is `passed`.

## v1 Requirements

### Foundation and Shell

- [x] **SHELL-01**: User can navigate the app on mobile with a bottom navigation shell that exposes only role-appropriate destinations and preserves current route access behavior. Outcome: validated in Phase 1.
- [x] **SHELL-02**: User can use shared mobile-first UI primitives for cards, status chips, buttons, tabs, bottom sheets, floating action bars, loading states, empty states, and error states consistently across changed screens. Outcome: validated in Phase 1.
- [x] **SHELL-03**: User can interact with changed mobile screens at 360-430px without bottom navs, floating CTAs, toasts, or sheets covering primary content or actions. Outcome: validated through source/build/browser evidence.
- [x] **SHELL-04**: User can switch Vietnamese/English language and see all new UI labels translated in both locales. Outcome: validated by i18n audit.
- [x] **SHELL-05**: Developer can verify route/access behavior, feature inventory, and no-regression checklists before shipping each refactor phase. Outcome: validated in phase artifacts.

### Guest Debt and Payment

- [x] **DEBT-01**: Guest can open the home page without login and immediately see debt as the primary content using data from `view_member_debt_summary`. Outcome: validated in Phase 1 and browser smoke.
- [x] **DEBT-02**: Guest can view each debtor's name, total debt, unpaid session count, loading state, empty state, and additional pages via the existing load-more behavior. Outcome: validated in Phase 1.
- [x] **DEBT-03**: Guest can open a member debt detail page from debt/home/member list and see debt history from `view_member_session_details`. Outcome: validated in Phase 1 and quick navigation follow-up.
- [x] **DEBT-04**: Guest can create a QR payment for one member or one unpaid session without frontend recalculating debt amounts. Outcome: validated in Phase 1 and payment smoke.
- [x] **DEBT-05**: Guest can select multiple debt members and create a group QR payment through the existing `create_group_payment` flow. Outcome: validated through source and non-mutating browser coverage.
- [x] **DEBT-06**: User can see QR payment amount, transfer content, copied code, member breakdown, and payment completion state consistently across home and member-detail entry points. Outcome: validated in Phase 1 and browser smoke.

### Session Detail Cockpit

- [x] **SESS-01**: Guest can open session detail in read-only mode while authenticated/admin users retain existing mutation controls according to current auth and session status gates. Outcome: validated in Phase 2.
- [x] **SESS-02**: User can view session overview details including title, date/time, court fee, shuttle fee, status, total collected when relevant, and cancelled/read-only messaging. Outcome: validated in Phase 2.
- [x] **SESS-03**: Admin can edit allowed session fields, cancel an open session, and finalize an open session using the existing Supabase contracts and status locks. Outcome: validated after audit closure; edit writes current `sessions` fields.
- [x] **SESS-04**: Admin can add active members to a session and remove registered members without changing `add_member_to_session_full_presence` or `session_registrations` behavior. Outcome: validated in Phase 2.
- [x] **SESS-05**: Admin can toggle interval-level attendance for every registered member and interval, with guest/read-only/done/waiting/absent disabled states preserved. Outcome: validated in Phase 2.
- [x] **SESS-06**: Admin can mark a registered member absent while preserving that state separately from member removal. Outcome: adjusted after schema merge; current source clears `interval_presence` rows and derives absent state from zero present intervals.
- [x] **SESS-07**: User can view live cost summary from `calculate_session_costs`, including member total, interval count, court fee, shuttle fee, and surplus fund. Outcome: validated in Phase 2.
- [x] **SESS-08**: User can view finalized payment snapshots for waiting-for-payment and done sessions, including must-pay amount, paid amount, payment status, interval count, court fee, shuttle fee, and surplus. Outcome: validated in Phase 2.
- [x] **SESS-09**: User can create single QR payments and authenticated/admin group QR payments from session snapshots using existing snapshot and `create_group_payment` behavior. Outcome: validated in Phase 2 and payment smoke.
- [x] **SESS-10**: Admin can record manual cash payment from session snapshots using the existing `add_manual_payment` flow, with non-admin access prevented. Outcome: validated by source/build and security audit.
- [x] **SESS-11**: User can rely on realtime updates, polling/manual refresh fallback, and payment completion refresh without duplicate timers or stale visible state after navigation. Outcome: validated after queued-refresh fix.
- [x] **SESS-12**: User can use the session detail screen as task-focused mobile sections for overview, attendance, costs, and payments while preserving all current session operations. Outcome: validated in Phase 2 and later UI reviews.

### Admin and Supporting Screens

- [x] **ADMIN-01**: User can view the sessions list read-only, see session cards with status/date/interval/registration counts, and navigate to session detail while create/mutation actions remain admin-only. Outcome: validated in Phase 3.
- [x] **ADMIN-02**: Admin can create a session with title, date, start time, end time, court fee, and shuttle fee through `create_session_with_bookings`, with end-time validation and loading/error/success feedback preserved. Outcome: validated after audit closure.
- [x] **ADMIN-03**: User can view the member list on mobile, and currently authorized users can add, edit, and delete members with display name, role, active status, and create-another behavior preserved. Outcome: validated after removing retired `is_permanent` usage.
- [x] **ADMIN-04**: User can log in, see login errors, log out, access profile/member link, see authenticated debt badge, and use header actions after the shell refactor. Outcome: validated in Phase 1.
- [x] **ADMIN-05**: User can use QR and manual payment modals as responsive mobile sheets or desktop dialogs without losing dialog semantics, close actions, QR alt text, or payment polling/refresh behavior. Outcome: validated in Phase 3 and UI reviews.
- [x] **ADMIN-06**: User can view table-to-card mobile layouts without losing financial fields, status fields, or actions that existed in desktop tables. Outcome: validated in Phase 3.

## Future Requirements

Deferred to future milestones:

- **DEBT-F01**: User can filter debt by overdue, high amount, partial payment, or due date after backend exposes reliable fields.
- **DEBT-F02**: Guest can submit "I transferred" confirmation for admin review after a backend confirmation workflow exists.
- **DEBT-F03**: User can select arbitrary unpaid sessions from member detail and create a partial group QR payment.
- **OPS-F01**: Admin can view activity/audit history for session, attendance, member, and payment changes.
- **OPS-F02**: User can use offline-friendly attendance capture when network is unstable.
- **OPS-F03**: Admin can see explicit realtime health or sync status beyond manual refresh/fallback behavior.
- **QA-F01**: Developer can run a full automated E2E suite for guest debt payment and admin session flows against stable test data.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Backend business logic rewrite | Existing Supabase RPCs/views own cost, session creation, payment grouping, and manual payment logic |
| Database schema change | UI refactor used current contracts; audit closure adapted frontend to existing schema |
| Frontend fee allocation | Violates backend boundary and risks wrong shared-cost behavior |
| Prototype-only overdue/late debt filters | Current frontend does not have reliable backend fields for these states |
| Guest manual transfer confirmation | Current behavior is QR polling and admin manual cash payment |
| Native mobile app | Milestone remains Vue SPA |
| Third-party UI framework migration | Current Tailwind/Vue stack is sufficient and lower risk |
